# PRD-011 — Passive predictive boiler-temperature diagnostics

Status: Active

Approved: 2026-07-21

## Summary

Add a fixed-allocation ESP32 temperature predictor that calculates filtered
temperature, slope, recent heater/pump command activity, and linear 5/10/20
second forecasts. Export predictions and hypothetical heater reductions through
history and CSV while leaving the existing heater controller unchanged.

## Goals

- Produce deterministic, versioned passive prediction diagnostics on ESP32-C3.
- Preserve the nonlinear ten-second duty controller as the sole heater policy.
- Fall back explicitly for invalid model, sensor, timing, bounds, or prediction.
- Retain ten minutes of enriched history and make it exportable for later
  validation.

## Non-Goals

- Active prediction-based heater control, PID replacement, runtime learning, or
  automatic tuning/model updates.
- Desktop training, fitting, simulation, or coefficient optimization.
- Scale, weight, flow, pressure, yield, flashing, deployment, or energized
  hardware authorization.

## Context

The current firmware has no PID; it uses a nonlinear temperature-error duty
curve and ten-second SSR window. PRD-011 treats that curve as the primary
controller. The checked-in fixed model is an unvalidated slope-extrapolation
seed suitable only for passive data collection.

This feature intentionally supersedes PRD-007's 12 KiB history budget,
sixty-sample page size, and unchanged CSV-column condition. It does not alter
PRD-007's pending Human/target acceptance state or its rule that API history is
observational and cannot feed control decisions.

## Requirements

- Use raw-temperature EMA filtering, a three-second slope, diagnostic
  acceleration, and fixed rolling command histories without heap allocation.
- Validate version, feature schema, checksum, coefficients, input bounds,
  sample timing, slope, and prediction sanity before marking a result usable.
- Evaluate separate raw-coefficient models for 5, 10, and 20 seconds and report
  their maximum as predicted peak.
- Calculate deadband/gain/hard-cutoff output as a hypothetical reduction only;
  it must never increase duty or reach the SSR command path.
- Keep 600 enriched RAM samples within 40 KiB and return at most eight per
  authenticated history page.
- Keep new history diagnostics optional for older firmware, persist them as
  nullable mobile data, and append stable CSV columns.
- Preserve every existing sensor, timeout, fault, over-temperature, cooldown,
  mutex, safety-lease, and fail-off boundary.

## Implementation Overview

1. Add pure model validation, filtering, history, inference, and fallback code.
2. Integrate passive diagnostics beside the existing controller and prove
   identical heater traces with prediction disabled.
3. Extend bounded history, protocol, mobile persistence, CSV, and documentation.
4. Complete host, contract, workspace, target-build, and supervised runtime
   evidence without enabling active correction.

## Acceptance Criteria

- [x] Valid fixed coefficients produce bounded 5/10/20-second predictions after
  thirty seconds of mature history.
- [x] Every invalid condition yields a specific fallback and nullable forecast.
- [x] Passive and disabled prediction produce identical heater command traces.
- [x] Ten-minute enriched history stays within its compile-time budgets and
  synchronizes through strict eight-sample pages.
- [x] Older history remains compatible; new diagnostics persist and export.
- [x] Host, sanitizer, protocol, simulator, mobile, and available target checks
  pass with evidence boundaries documented.
- [ ] Connected-target resource/timing and passive physical validation remain
  explicit Human gates.

## Open Questions

  validated coefficients and physical acceptance thresholds.
