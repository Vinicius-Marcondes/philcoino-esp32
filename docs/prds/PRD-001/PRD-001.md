# PRD-001 — Local espresso machine monitoring and temperature control

Status: Ready

Approved: 2026-07-03

## Summary

Build an iPhone application and ESP32-C3 HTTP interface that discover one espresso machine on local Wi-Fi, authenticate with a user-entered bearer token, monitor its temperature-control state, configure brew and steam targets, and select the active temperature mode.

## Goals

- Discover the ESP32 without scanning every local IP address.
- Pair one machine through explicit device selection and token entry.
- Reconnect reliably after app restarts and DHCP address changes.
- Display current temperatures, active mode, readiness, heater activity, connection state, and faults.
- Change validated brew and steam targets and persist them on the ESP32.
- Switch firmware control between brew and steam modes with safe defaults and timeout behavior.
- Keep mobile, protocol, simulator, and ESP-IDF firmware documentation in one repository.

## Non-Goals

- Provisioning the ESP32 onto Wi-Fi.
- Cloud accounts, remote internet access, or multiple saved machines.
- Remote activation of machine power, pump, water flow, or physical brewing.
- HTTPS for the accepted private-LAN v1 threat model.
- Making the phone part of the real-time temperature-control or safety loop.
- Resolving final mains wiring, regulatory compliance, or pressure-vessel protection in this PRD.

## Context

The ESP32-C3 Super Mini is powered only while the espresso machine is on. It reads two MAX6675 thermocouples, drives a heater through a FOTEK SSR-40 DA, and displays local information on a 128×32 SSD1306 OLED. The base thermocouple is the brew control sensor; the top thermocouple is the steam control sensor.

The mobile project uses Expo 54, React Native 0.81, TypeScript, and Bun. Zod will validate received payloads. A Bun/Hono simulator may implement the device contract for development without physical hardware. The shared API source of truth will be OpenAPI rather than TypeScript code so the C++ firmware and mobile app can follow the same contract.

The original machine over-temperature fuse or thermostat is planned to remain as an independent cutoff. Its exact trip behavior and series wiring, along with relay thermal verification, are recorded in `docs/side-notes.md` and remain outside approval for energized hardware.

## Requirements

### Repository and contract

- Organize the repository into mobile app, ESP-IDF firmware, protocol, simulator, and documentation boundaries.
- Define the HTTP API in a versioned OpenAPI 3.1 contract.
- Validate mobile and simulator payloads with Zod; validate all firmware inputs independently in C++.

### Discovery and pairing

- Firmware advertises `_philcoino._tcp.local` using mDNS with stable device ID, friendly name, API version, firmware version, model, and port.
- The app lists discovered devices and also supports manual IP entry.
- After selection, the user enters a bearer token; the app verifies it against an authenticated endpoint.
- The app securely persists the token, stable device ID, and last successful address for one machine.
- Startup tries the cached address first and rediscovers the stable device ID when necessary.

### API and monitoring

- `/healthz` and `/api/v1/device` are available without authentication.
- State and mutation endpoints require the bearer token over local HTTP port 80.
- The app polls `GET /api/v1/state` once per second while the primary machine screen is active.
- Firmware reports `heating`, `ready`, or `fault`; the app separately reports connecting, online, unauthorized, not found, and offline states.
- `ready` requires the active control temperature to remain within ±1°C of target for three continuous seconds.
- Firmware reports both temperatures, both targets, active mode, heater activity, fault data, steam timeout, and uptime.

### Temperature and mode control

- Brew target accepts whole values from 85°C through 95°C inclusive.
- Steam target accepts whole values from 110°C through 120°C inclusive.
- Temperature changes require explicit confirmation and are persisted in ESP32 NVS.
- The app switches active mode through an idempotent `PUT /api/v1/mode` request.
- ESP32 always boots into brew mode; active mode is not persisted.
- Steam timeout begins when steam temperature first qualifies as ready and returns the ESP32 to brew mode after five minutes.
- The timeout continues despite later temperature dips or loss of app connectivity.
- The app displays mode changes only after firmware acknowledgement.

### Firmware safety behavior

- ESP32 owns sampling, heater control, validation, timeouts, and fault handling without depending on app connectivity.
- Open thermocouple, invalid measurement, over-temperature, heating timeout, or internal control failure immediately de-energizes the SSR command.
- Faults use stable codes. Non-over-temperature faults remain latched until machine power is cycled; over-temperature remains latched until power cycle or explicit dismissal after cooldown to the active target.
- Both thermocouples are monitored in all modes even though each mode has one primary control sensor.

## Implementation Overview

1. Establish the proposed monorepo layout and version-pinned documentation.
2. Finalize OpenAPI schemas, examples, error codes, and safety constants.
3. Build the ESP-IDF sensor, persistence, control-state, mDNS, OLED, and HTTP boundaries.
4. Build a Hono simulator against the same contract.
5. Build mobile discovery, pairing, secure persistence, polling, dashboard, settings, and mode control.
6. Validate contract behavior with simulator tests, firmware integration tests, and supervised physical-hardware tests.

## Acceptance Criteria

- [ ] An iPhone on the same Wi-Fi discovers the ESP32 through `_philcoino._tcp.local`.
- [ ] Manual IP entry can pair the device when mDNS discovery fails.
- [ ] Invalid tokens are rejected without saving a working connection.
- [ ] A valid token and selected device survive app restarts in secure local storage.
- [ ] A changed DHCP address is recovered by stable-device-ID rediscovery.
- [ ] The main screen updates state once per second and distinguishes offline from firmware-reported statuses.
- [ ] Malformed firmware responses are rejected by runtime schemas and shown as a connection/protocol error.
- [ ] Brew values outside 85–95°C and steam values outside 110–120°C are rejected by both app and firmware.
- [ ] Confirmed valid targets persist across ESP32 power cycles.
- [ ] Every ESP32 boot starts in brew mode.
- [ ] Switching mode is reflected only after ESP32 acknowledgement.
- [ ] Steam mode returns to brew five minutes after first becoming ready, including when the app disconnects.
- [ ] Ready status requires ±1°C stability for three continuous seconds.
- [ ] Open or invalid thermocouple data de-energizes the SSR command and latches `sensor_failure`.
- [ ] All defined faults de-energize the SSR command. Non-over-temperature faults remain latched until power cycle; over-temperature may be dismissed only after the active control temperature has returned to target.
- [ ] The app cannot remotely start brewing, activate the pump, or power the machine.
- [ ] API examples pass the shared contract and mobile Zod validation.

## Open Questions

- What exact ESP-IDF release will the firmware pin?
- What are the validated over-temperature, heating-timeout, and cross-sensor disagreement thresholds?
- What friendly device name and stable-ID provisioning mechanism will firmware use?
- What exact OLED breakout address, reset wiring, and I2C pin assignment will be used?
- Is direct GPIO10 drive reliable at the FOTEK input's 3 V lower boundary, or should a transistor/MOSFET switch the 5 V rail into the SSR input?
- What heat sink and enclosure thermal design does the relay require at the measured heater current and duty cycle?
- What visual dashboard and temperature-setting interaction will receive human design approval?
