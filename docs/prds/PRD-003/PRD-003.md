# PRD-003 — Steam temperature offset compensation

Status: Ready

Approved: 2026-07-14

## Summary

Apply a fixed, firmware-configured `+5°C` offset to the valid boiler-base
thermocouple reading while Steam mode is active. The corrected temperature is
the authoritative Steam temperature for heater control, readiness, timeouts,
safety limits, API state, and OLED presentation. Brew mode continues to use and
report the unmodified sensor reading.

The offset is a compile-time firmware constant in the existing configuration
header. Changing it requires editing the source, rebuilding, and reflashing the
firmware; it is not dynamically configurable or persisted on the device.

## Goals

- Compensate for the owner-observed temperature difference between the
  boiler-base sensor and the top of the boiler during steaming.
- Stop Steam heating at a raw sensor reading of approximately `115°C` when the
  configured Steam target is `120°C`.
- Use one corrected Steam temperature consistently across control, safety,
  protocol, and local display behavior.
- Keep the correction easy to adjust manually in the existing firmware
  configuration source.
- Preserve firmware authority over temperature control and safety behavior.

## Non-Goals

- Runtime, NVS, API, OLED-menu, simulator-control, or mobile configuration of
  the offset.
- Automatic calibration, scaling, interpolation, or a temperature-dependent
  correction curve.
- Applying an offset to Brew mode.
- Adding another sensor or reporting both raw and corrected temperatures.
- Changing Brew or Steam target ranges, target persistence, or API shapes.
- Treating the `5°C` observation or this software correction as proof of
  calibrated boiler temperature, heater safety, or safe energized operation.

## Context

Philcoino controls both Brew and Steam modes from one MAX6675 thermocouple at
the base of the boiler. The owner reports that an independently observed
temperature near the top of the boiler is approximately `5°C` higher than the
base sensor reading around `110°C`, `115°C`, and `120°C`. Consequently, when
the current firmware reports and controls to `120°C`, the observed upper-boiler
temperature may already be `125°C` or higher.

This PRD adopts a fixed `+5°C` Steam correction as an owner-selected
compensation. The observation still requires repeatable instrumented physical
validation across warm-up, steady state, and recovery. Existing independent
thermal-cutoff, single-sensor, SSR, timing, and energized-test limitations in
`docs/SAFETY.md`, `docs/side-notes.md`, and `CODEBASE_REVIEW_REPORT.md` remain
unresolved.

## Requirements

### Configuration and temperature semantics

- Add one compile-time firmware configuration constant named for the Steam
  temperature offset with a value of exactly `5°C`.
- Keep the constant in the existing `firmware_config` configuration header
  alongside the other temperature-control constants.
- Do not load, persist, mutate, or expose the offset through NVS, HTTP, mDNS,
  the OLED, or the mobile app.
- Validate the raw thermocouple status and finite numeric reading before
  applying the offset. An invalid, open, non-finite, or transport-failed sample
  must still latch `sensor_failure` and command the heater off.
- Define the active control temperature as the raw thermocouple reading in Brew
  mode and the raw reading plus `5°C` in Steam mode.
- Apply the correction once only; no caller may add the offset independently.

### Steam control and safety behavior

- Use the corrected active temperature for every Steam-mode decision,
  including heater demand, duty calculation, recovery behavior, readiness,
  heating timeout demand tracking, and the start of the five-minute Steam
  timeout.
- At a Steam target of `120°C`, a valid raw reading of `115°C` must produce an
  active temperature of `120°C` and no request for additional heater power.
- Use the corrected active temperature for the Steam over-temperature threshold
  and over-temperature dismissal checks. With the existing `130°C` threshold,
  a raw Steam reading of `125°C` must latch `over_temperature`.
- Keep the existing Steam targets, target validation, readiness band, stability
  duration, heating timeout, Steam return timeout, and over-temperature
  threshold numerically unchanged.
- Switching modes must reset the existing readiness, timeout, recovery, and
  heater-window state as it does today; Brew must never inherit the Steam
  correction.

### API, OLED, mobile, and simulator behavior

- In Steam mode, firmware snapshots and API responses must expose the corrected
  value as `boilerTemperatureC`. The raw reading is not separately exposed.
- The OLED must display the same corrected Steam temperature returned by the
  API. Mobile must continue displaying the acknowledged API value without
  adding its own correction.
- In Brew mode, `boilerTemperatureC` and the OLED must continue to show the raw
  sensor reading.
- A mode change may therefore change the reported temperature by `5°C` even
  when the physical sensor sample has not changed. Document this semantic
  behavior in the OpenAPI field description and human-facing architecture and
  tuning documentation without changing the API schema.
- The deterministic simulator may continue treating its configured
  `boilerTemperatureC` as the already-effective logical control temperature. It
  must not claim to simulate the physical top-to-bottom boiler gradient or
  validate the firmware correction.

### Verification and safety evidence

- Add firmware configuration and controller host tests covering both modes and
  preventing accidental double application of the offset.
- Cover Steam heater cutoff, readiness stability, Steam-timeout start, recovery
  calculations, heating-demand tracking, over-temperature latching, dismissal,
  API serialization, and OLED presentation at offset-relevant boundaries.
- Verify that Brew results remain unchanged for the same raw readings.
- Run the firmware host suite and contract-capture validation, and run the
  pinned ESP-IDF target build when its toolchain is available.
- Treat host tests, protocol tests, simulator tests, and target builds as
  software evidence only. Any claim that `5°C` accurately represents the
  physical gradient requires a separately approved, instrumented physical
  validation with the existing safety constraints preserved.

## Implementation Overview

1. Clarify the mode-dependent `boilerTemperatureC` semantics in the OpenAPI
   contract before changing consumers or firmware behavior.
2. Add the fixed Steam offset to `firmware_config` and centralize raw-to-active
   temperature conversion inside the temperature controller.
3. Route all Steam control, safety, API snapshot, and OLED presentation paths
   through that single corrected active temperature.
4. Add boundary-focused firmware tests and update architecture, development,
   tuning, safety, and deferred physical-validation documentation.
5. Complete software verification, then retain physical calibration as explicit
   human acceptance work rather than inferring it from passing tests.

## Acceptance Criteria

- [ ] Firmware defines one compile-time Steam offset constant with the value
  `5°C` in the existing configuration header.
- [ ] The offset cannot be changed through the API, mobile app, OLED, NVS, or
  simulator controls.
- [ ] A valid raw `115°C` reading in Steam mode is controlled, returned by the
  API, and displayed on the OLED as `120°C`.
- [ ] At a `120°C` Steam target and raw `115°C` reading, firmware commands no
  additional heat and can become ready only after the existing continuous
  three-second stability requirement.
- [ ] Steam readiness starts the existing five-minute timeout using the
  corrected temperature.
- [ ] Steam duty, recovery, and heating-timeout demand calculations use the
  corrected temperature exactly once.
- [ ] With the existing Steam safety limit, a valid raw `125°C` reading is
  treated as corrected `130°C`, latches `over_temperature`, and commands the
  heater off.
- [ ] Over-temperature dismissal uses the corrected Steam temperature and
  retains all existing validity and cooldown requirements.
- [ ] The same valid raw reading remains unmodified in Brew control, API, and
  OLED behavior.
- [ ] Switching between Brew and Steam resets the existing mode-sensitive state
  and may change the reported temperature by exactly `5°C` without changing
  the underlying sample.
- [ ] Invalid or failed raw sensor samples latch `sensor_failure` before any
  correction is used.
- [ ] The API schema, target ranges, persisted targets, and mobile mutation
  workflows remain backward compatible.
- [ ] Firmware host tests and contract captures pass, and the pinned ESP-IDF
  target build result is recorded when available.
- [ ] Documentation labels the `5°C` value as owner-selected pending repeatable
  physical validation and does not present software evidence as heater-safety
  approval.

## Open Questions

There are no open questions blocking the software scope. The owner has selected
a fixed `+5°C` Steam offset, applied consistently to control, safety, API, and
OLED behavior, with no dynamic configuration or scaling.

The following physical-validation work is intentionally deferred and does not
change the implementation decision:

- A separately approved test should place an independent, suitable reference
  temperature instrument near the top of the boiler and record it alongside
  the raw boiler-base sensor reading. The instrument, mounting position,
  firmware build, boiler state, and test conditions must be recorded.
- Paired readings should be collected repeatedly near raw sensor values of
  `110°C`, `115°C`, and `120°C`, including temperature rise, steady Steam
  operation, and recovery after a temperature drop. This determines whether
  the physical difference is repeatable; software and simulator tests cannot
  establish it.
- Boiler fill, pressure, ambient conditions, and heat-soak duration should be
  recorded because they may affect the observed gradient. This PRD does not
  require the difference to be proven constant before implementing the
  owner-selected offset.
- If later physical evidence shows that a fixed `+5°C` correction is not
  suitable, do not introduce scaling or a correction curve under this PRD.
  Revisit the constant or define a separate calibration PRD.
- This deferred procedure does not authorize energized testing. It remains
  subject to the independent cutoff, SSR, wiring, enclosure, instrumentation,
  supervision, and approval boundaries in `docs/SAFETY.md`.
