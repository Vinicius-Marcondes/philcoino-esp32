# PRD-004 — Extraction temperature compensation and cooldown workflow

Status: Complete — Human Accepted 2026-07-16

Approved: 2026-07-14

## Summary

Add two firmware-owned thermal workflows to API v2 and the mobile Dashboard.
Extraction can start only while the machine is in Brew mode. During Manual or
profile main extraction, firmware temporarily applies a fixed `+2°C` bias only
to the Brew heater-duty calculation so heating can anticipate flow-related heat
loss. Profile pre-infusion has a separate fixed bias initially set to `0°C`,
and soak never applies compensation.

Add an acknowledged cooldown action that switches the machine to Brew, inhibits
the heater without changing the user's heater permission, and commands the pump
until the Brew target is reached or 45 seconds elapse. The pump then stops and
the heater remains inhibited for a fixed five-second stabilization period before
normal Brew control may resume.

Firmware remains authoritative for phase eligibility, temperature validation,
output commands, deadlines, faults, and recovery. Pump and heater states remain
command states only and do not prove physical flow, current, cooling, or
de-energization.

## Goals

- Anticipate the predictable Brew temperature drop during the water-intensive
  part of an extraction without adding runtime tuning or profile complexity.
- Apply compensation to Manual and profile main extraction while keeping
  pre-infusion independently configurable and initially disabled.
- Prevent extraction from starting in Steam and prevent entering Steam while an
  extraction is active.
- Provide a prominent, confirmed Dashboard action for automatic cooldown with
  an always-available Stop action and firmware-owned limits.
- Preserve acknowledged mobile state, deterministic simulator behavior, API v1
  compatibility, and firmware authority over both physical commands.

## Non-Goals

- User-configurable compensation values, profile-specific heat settings, or
  persisted thermal curves.
- Pressure, flow, water-level, pump-current, SSR-output, or physical-switch
  sensing.
- PID control, arbitrary heater-duty recipes, or automatic tuning from prior
  extractions.
- Requiring Brew readiness or an enabled heater to run an extraction after its
  Brew-mode eligibility check succeeds.
- Using cooldown as fault dismissal, dry-boil protection, proof of water flow,
  or proof that the machine reached a physically calibrated temperature.
- Resuming extraction or cooldown after reset or power loss.
- Treating simulator, host, target-build, or software evidence as energized
  heater/pump safety evidence.

## Context

PRD-002 introduced firmware-owned Manual and timed-profile extraction with
independent pump and temperature controllers. It currently permits extraction
in Brew or Steam and keeps heater faults independent from pump execution.
PRD-004 intentionally supersedes only the mode-eligibility portion: new
extractions require acknowledged Brew mode, and Steam cannot be selected during
an active extraction. Readiness and heater faults continue not to stop an
otherwise eligible extraction; they only make temperature compensation
inactive when heating is not permitted safely.

The current temperature controller reacts to a drop using normal and recovery
curves. Because firmware now knows the exact Manual start and profile phase
boundaries, it can apply a bounded phase-aware bias before the single
boiler-base sensor observes the full effect of water flow. The `+2°C` value is
an owner-selected initial hypothesis, not a calibrated result.

Cooldown coordinates pump and heater behavior but still has no feedback that
water is flowing. The original series pump switch may prevent flow while the
firmware command reports `running`, and either SSR may fail independently.
Therefore cooldown requires a strict pump-command cutoff and must never be
presented as confirmed cooling or de-energization.

## Requirements

### Extraction eligibility and compensation

- Firmware must reject Start while the acknowledged control mode is Steam with
  a strict API v2 conflict response. Mobile must disable Start and explain that
  the user must switch to Brew, but firmware enforcement is authoritative.
- Firmware must reject a request to enter Steam while extraction is active.
  Starting or remaining in Brew is allowed without requiring readiness.
- Manual extraction is compensation-eligible from its acknowledged Start until
  Stop, cutoff, completion, reset, or failure.
- A profile applies its independent pre-infusion bias only during pre-infusion,
  applies no compensation during soak, and applies the main bias only during
  main extraction. The initial compile-time values are:

  - pre-infusion bias: `0°C`;
  - Manual/main-extraction bias: `+2°C`.
- The compensated heater-control target must be calculated as
  `min(brewTargetC + 2°C, brewOverTemperatureC - 1°C)` for Manual/main, with
  the corresponding `0°C` pre-infusion value. The bias affects only heater
  demand and duty; it must not mutate or persist the Brew target, change
  readiness semantics, reset or extend safety deadlines, change API/OLED target
  presentation, or alter over-temperature limits.
- Compensation must stop at the exact phase/workflow boundary and whenever a
  fault, disabled heater permission, failed output command, or other existing
  heater fail-off condition prevents safe heating. Existing faults and limits
  always override the bias.
- A temperature/heater fault that occurs during extraction must retain
  PRD-002's pump independence: it suppresses compensation and commands the
  heater off but does not itself stop the extraction pump workflow.
- API v2 state and OLED/mobile presentation must expose acknowledged
  compensation activity without presenting the fixed constants as runtime
  settings or implying measured thermal effectiveness.

### Cooldown workflow

- Provide authenticated API v2 Start and Stop operations plus cooldown state in
  the combined machine snapshot. Start must use a client idempotency key so a
  retry cannot restart the 45-second deadline; Stop must be idempotent.
- Cooldown Start is allowed only with no active extraction or cooldown, a valid
  boiler reading, no latched machine fault, and a Brew-effective temperature
  above the current Brew target. Invalid starts must return strict,
  distinguishable conflicts.
- An acknowledged Start must atomically establish the firmware workflow,
  switch control mode to Brew, apply a transient heater inhibit, command the
  heater off, snapshot the Brew target used as the completion threshold, and
  then command the pump running. A heater-off command failure must prevent the
  pump from starting.
- The transient inhibit must remain independent from the user's volatile heater
  permission. Cooldown must never turn a manually disabled heater permission
  on. If permission is enabled, it may take effect only after successful
  completion of stabilization.
- The pumping phase ends when the validated Brew temperature is less than or
  equal to the snapshotted Brew target, 45 seconds have elapsed, or Stop is
  acknowledged. Each path must command the pump off immediately and enter a
  five-second stabilization phase with the heater still inhibited.
- After five uninterrupted stabilization seconds, firmware may remove the
  transient inhibit and resume normal Brew control according to the current
  heater permission. The pump remains off.
- A sensor fault, pump-command failure, heater-command failure, reset, or power
  loss must abort cooldown and command both outputs off where firmware can do
  so. Fault paths must not automatically restore heating or dismiss a fault.
- Extraction Start and profile replacement must conflict with active cooldown;
  cooldown Start must conflict with active extraction; entering Steam must
  conflict with either active workflow. Reset and power loss must boot with no
  active cooldown and must never restore a running pump command.
- Phone disconnection or app backgrounding must not alter an acknowledged
  cooldown. Firmware owns temperature checks, the 45-second cutoff, the
  five-second stabilization period, and all output transitions.
- Cooldown state must distinguish at least idle, pumping, and stabilizing, show
  acknowledged elapsed/remaining timing and pump command, and distinguish
  target-reached, cutoff, Stop, and failure outcomes where applicable.

### Mobile experience

- Place a prominent `Cooldown machine` action on the Dashboard. Before Start,
  require confirmation that shows the current acknowledged temperature, the
  snapshotted Brew target, possible water use, the 45-second pump-command limit,
  and the absence of physical flow feedback.
- Keep a prominent Stop action available throughout pumping. During
  stabilization, show that the pump command is off and normal heating remains
  temporarily inhibited.
- Disable or reject conflicting extraction, profile-export, and Steam actions
  with actionable explanations. Extraction Start in Steam must direct the user
  to switch to Brew rather than switching silently.
- Show compensation, cooldown phase, timing, completion/cutoff reason, and
  output commands only from validated device acknowledgements. Requested values
  must never appear as live state before acknowledgement.
- Preserve serialized mutations, polling cancellation/generation semantics,
  disconnect clearing, retry idempotency, localization, accessibility, Dynamic
  Type, and non-color-only status communication.
- The exact confirmation hierarchy, prominent Stop treatment, and Dashboard
  placement require owner review in debug mode before final visual acceptance.

### Contract, simulator, firmware, and safety

- Define additive API v2 changes first in `packages/protocol/openapi.yaml`, then
  align strict Zod schemas, fixtures, simulator, mobile parsing, independent C++
  parsing/serialization, captures, and documentation. API v1 paths and shapes
  remain unchanged.
- Keep compensation and cooldown constants compile-time and firmware-owned. No
  profile, NVS, mobile storage, or API mutation may configure them.
- Model cooldown and compensation deterministically in the simulator for
  contract/UI testing while stating that simulator temperature movement is not
  firmware timing or physical thermal evidence.
- Coordinate temperature and extraction state inside firmware with bounded,
  deadlock-safe synchronization. HTTP, display, persistence, or cross-domain
  lock ordering must not extend heater pulses, pump deadlines, cooldown cutoff,
  or stabilization timing.
- Preserve the heater safety lease, pump fail-off abstraction, monotonic
  wrap-safe timing, single-owner output commands, and boot-off ordering.
- Update tuning, architecture, protocol, safety, side-note, and review documents
  without weakening unresolved single-sensor, cutoff, SSR, timing, security,
  wiring, enclosure, or physical-output findings.

## Implementation Overview

1. Extend the authoritative API v2 contract with Brew-mode conflicts,
   acknowledged compensation state, and idempotent cooldown operations/state.
2. Add pure deterministic compensation and cooldown policies, then integrate
   them through a bounded firmware coordinator that preserves independent
   real-time output deadlines and existing fail-off ownership.
3. Add deterministic simulator behavior and connect the serialized mobile
   mutation/polling flow, including debug-mode visual review of confirmation,
   active, Stop, stabilization, cutoff, failure, and Steam-blocked states.
4. Align OLED and public documentation, run all affected software/target checks,
   and keep physical thermal tuning and energized validation separately
   authorization-gated.

## Acceptance Criteria

- [x] API v1 remains unchanged while API v2 strictly exposes compensation and
  cooldown state, idempotent cooldown Start/Stop, and distinguishable workflow
  conflicts.
- [x] Mobile blocks and firmware rejects extraction Start in Steam; firmware
  also rejects entering Steam during extraction or cooldown.
- [x] Manual and profile main extraction apply exactly the fixed `+2°C`
  heater-duty target bias, pre-infusion applies `0°C`, and soak applies no
  compensation at exact monotonic phase boundaries.
- [x] The compensated duty target is clamped below the Brew over-temperature
  threshold and does not change persisted targets, readiness, timeouts,
  over-temperature limits, API/OLED targets, or profile data.
- [x] Disabled heater permission and every existing heater fault/fail-off rule
  override compensation, while a temperature fault does not independently stop
  an otherwise active extraction pump workflow.
- [x] Cooldown Start requires valid eligible state, switches to Brew, commands
  heater off before pump running, snapshots the Brew threshold, and never
  changes the user's heater permission.
- [x] Cooldown commands the pump off at the first validated sample at or below
  the Brew target, at exactly 45 seconds, or on acknowledged Stop, then keeps
  both commands off through exactly five stabilization seconds.
- [x] Normal Brew control resumes after stabilization only when the current
  heater permission allows it; a previously disabled heater remains disabled.
- [x] Cooldown and extraction/profile replacement are mutually exclusive;
  same-key retries preserve elapsed time, competing starts conflict, Stop is
  idempotent, and app disconnection does not extend or interrupt firmware timing.
- [x] Sensor/output failures, reset, and power loss command outputs off where
  possible, do not restore cooldown/extraction at boot, and never report
  physical flow, cooling, current, SSR state, or de-energization as confirmed.
- [x] Dashboard confirmation, prominent Stop, active/stabilizing/cutoff states,
  Steam guidance, localization, VoiceOver, and large Dynamic Type receive
  explicit owner visual approval using acknowledged debug states.
- [x] OpenAPI validation, protocol/simulator/mobile tests and typechecks, mobile
  lint/config/export checks, strict C++ host tests and captures, and the pinned
  ESP-IDF target build pass for the affected scope or unavailable checks are
  explicitly recorded.
- [x] Human acceptance was recorded on 2026-07-16 from the owner's report that
  all implemented features and technical-equipment energy-control checks passed
  for the tested configuration. Raw build/setup/instrument records were not
  committed, so this is not certification or a general safety guarantee.

## Open Questions

There are no open questions blocking PRD-004. On 2026-07-16, the owner accepted
the implemented `0°C` pre-infusion bias, `+2°C` main/Manual bias, Brew-target
cooldown threshold, five-second stabilization, and final Dashboard experience
after reporting functional and technical-equipment testing. Any later constant,
runtime tuning, per-profile behavior, or architecture change requires a new
product decision.
