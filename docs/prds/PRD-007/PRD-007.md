# PRD-007 — ESP32 rolling temperature history and mobile backfill

Status: Active

Approved: 2026-07-18

## Summary

Add a volatile, bounded ESP32 history buffer that records full graph context
and automatically backfills the phone's current-day SQLite history after
connection, foregrounding, or reconnection. Keep live state and controls usable
independently from this observational synchronization.

Change the detailed Live graph from sixty-second to horizontally paged
thirty-second windows. Open on the newest window, follow new samples only while
the user remains there, and preserve access to older windows without a visible
scrollbar.

## Goals

- Fill graph gaps for app or network interruptions of up to ten minutes while
  the ESP32 remains powered.
- Recover temperature, targets, mode, heater/pump command, status, and fault
  context without duplicates in graph history or CSV export.
- Keep history capture, transfer, and mobile persistence outside every control,
  mutation, safety, and physical-state authority boundary.
- Preserve API v1 and all existing API v2 response shapes while adding a strict
  authenticated history endpoint.
- Preserve explicit gaps after reboot, retention overflow, or unavailable data.

## Non-Goals

- Background mobile polling, cloud synchronization, interpolation,
  notifications, or history across prior local days.
- Persisting history in ESP32 NVS/flash or resuming it after power loss.
- Using history to control temperature, targets, readiness, faults, heater, or
  pump behavior.
- Presenting heater/pump command state as proof of physical operation,
  de-energization, flow, or cooling.
- Installing a new dependency or changing existing API v1/v2 state and mutation
  payloads.

## Context

PRD-006 persists validated foreground polls and deliberately renders background
or offline periods as graph gaps because mobile operating systems do not
guarantee one-second local HTTP work in the background. The firmware already
produces authoritative temperature/control snapshots while the app is absent,
so a small RAM-only device history can recover short gaps without moving any
safety responsibility to the phone.

The sensor/control path updates every 500 ms and the Dashboard observes state at
approximately one-second completion-driven intervals. A one-Hertz history
therefore matches current graph fidelity while bounding ESP32-C3 RAM and HTTP
response work.

## Requirements

### Firmware history and safety boundary

- Record one compact sample per second in a fixed 600-sample RAM ring buffer.
  The complete buffer plus metadata must use no more than 12 KiB of static RAM.
- Give each boot an ephemeral 128-bit identifier and each actual sample an
  increasing sequence plus 64-bit monotonic uptime. Reset all history on boot.
- Record boiler temperature, brew/steam targets, active mode, heater permission
  and command, the fail-off pump wrapper's command, machine status, and nullable
  fault code from acknowledged firmware state.
- If sampling is delayed, record only the current state and leave elapsed time
  explicit; never synthesize missed samples.
- A history writer must never wait on network work. Copy at most sixty samples
  under a separate bounded history lock, serialize after unlocking, and allow
  observational samples to be skipped rather than delay the control loop.
- Do not add NVS writes, change workflow synchronization, extend safety
  deadlines, or introduce a new dependency.

### Additive API v2 contract

- Add authenticated `GET /api/v2/history`, with authentication resolved before
  query parsing. Existing API v1 and API v2 payloads remain unchanged.
- Accept either no cursor or the exact pair `bootId` and `afterSequence`.
  Reject unknown, duplicate, partial, malformed, or impossible cursor values
  with the existing strict API v2 malformed-request behavior.
- Return ascending pages of at most sixty samples with device/boot identity,
  capture uptime, oldest/latest available sequence, next cursor, `hasMore`, and
  `initial`, `continuous`, `truncated`, or `reset` continuity.
- A missing cursor starts at the oldest available sample. An evicted cursor
  returns retained data as `truncated`; a different boot ID returns new-boot
  data as `reset`.
- Define the endpoint first in OpenAPI, mirror it in strict Zod schemas and
  fixtures, and validate/serialize it independently in firmware C++.

### Mobile synchronization, persistence, and presentation

- Start an abortable history session only after a fresh combined state restores
  normal live behavior. Backgrounding cancels it; the next connection or
  foreground transition may retry.
- Treat endpoint `404` as unsupported older firmware and retain foreground-only
  history. Other failures show a localized graph-scoped warning without
  blocking polling, controls, or mutations.
- Show a subtle localized restoring status and publish each page only after its
  durable SQLite transaction completes.
- Anchor device uptime to phone UTC using the first page request/response
  midpoint and use that anchor for the complete synchronization batch.
- Persist source boot/sequence and the durable cursor. Enforce uniqueness for
  device samples and replace overlapping phone-originated rows from the same
  boot/uptime interval.
- Preserve current-local-day pruning and existing CSV columns. Recovered rows
  participate in Live, Today, downsampling, and export like foreground rows.
- Use consecutive thirty-second Live windows. Open and follow the newest page;
  do not pull a user away from an older page when new samples arrive. Keep the
  Today overview unchanged and keep horizontal scrolling hidden-scrollbar.
- Treat device identity, boot changes, uptime/timestamp discontinuities, and
  truncated history as explicit graph boundaries; never interpolate them.

### Simulator, compatibility, and evidence

- Give deterministic simulator time the same one-Hertz capture, 600-sample
  retention, cursor pagination, overflow, reset, and state-transition behavior.
- Keep debug and older-firmware paths functional without claiming recovered
  samples when the endpoint is unavailable.
- Update architecture, API outline, development, safety/evidence, and public
  behavior documentation with the implemented boundary.
- Treat protocol, simulator, mobile, host, and target-build results as software
  evidence only. Human visual/native review and target runtime resource/timing
  evidence remain separate completion gates.

## Implementation Overview

1. Define and validate the additive strict history contract.
2. Add deterministic simulator history and integration coverage.
3. Add the compact host-tested firmware ring, authenticated endpoint, and
   bounded ESP-IDF synchronization.
4. Add mobile cursor synchronization, SQLite migration/merge, and graph states.
5. Change Live paging to thirty seconds, align documentation, and complete
   automated plus Human/target acceptance.

## Acceptance Criteria

- [x] Interruptions of two, five, and ten minutes backfill without artificial
  graph gaps; longer interruptions, reboot, and overflow remain explicit gaps.
- [x] Recovered samples render and export full graph context without duplicate
  rows or changed CSV columns.
- [x] Live state and controls remain usable during history synchronization,
  after cancellation/failure, and with older unsupported firmware.
- [x] The authenticated strict history endpoint paginates at sixty samples,
  distinguishes initial/continuous/truncated/reset continuity, and leaves all
  existing wire shapes unchanged.
- [ ] Firmware history stays within its fixed RAM/page budgets and does not
  delay control-loop work or add flash persistence.
- [x] Live opens on the newest thirty-second window, remains horizontally
  pageable, follows only while viewing latest, and keeps Today unchanged.
- [ ] Protocol, simulator, mobile, firmware host/sanitizer/capture, and pinned
  ESP-IDF target-build checks pass for the affected areas.
- [ ] Human accepts iPhone paging, older-window inspection during live updates,
  minimize/restore, force-close/reopen, status/error presentation, reboot gaps,
  and CSV output.
- [ ] Target evidence records image/static RAM plus representative history
  request heap/stack and confirms no material control/network timing regression.
