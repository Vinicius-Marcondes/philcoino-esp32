# PRD-002 — Pump and extraction profile control

Status: Ready

Approved: 2026-07-12

## Summary

Add local mobile control of espresso extraction through a second active-high
FOTEK SSR-40 DA connected to ESP32-C3 GPIO10. The app can run a built-in Manual
mode or an exported timed profile, while firmware owns pump timing and completes
an acknowledged extraction without depending on the phone connection.

The machine's original pump switch remains electrically in series, normally on,
and can interrupt pump power locally. Because firmware does not sense that switch
or pump current, the system reports only the ESP32 pump command, never confirmed
physical pump operation or de-energization.

## Goals

- Start and stop extraction from the primary mobile screen.
- Support Manual extraction and up to four named timed profiles.
- Support optional timed pre-infusion, soak, and main extraction phases.
- Persist profiles independently on mobile and ESP32, with explicit whole-set
  export from mobile to machine.
- Keep extraction timing, limits, and pump fail-off behavior authoritative in
  firmware after Start acknowledgement.
- Preserve API v1 during migration to an expanded API v2 contract.

## Non-Goals

- Pressure, flow, water-level, dose, weight, or physical-switch sensing.
- Variable pump power, pressure profiling, or arbitrary multi-stage recipes.
- Automatic profile synchronization or conflict merging.
- Resuming an interrupted extraction after reset or power loss.
- Making the phone part of the extraction timing or safety loop.
- Treating simulator, software, or existing operation as proof of mains safety.

## Context

Philcoino currently controls boiler temperature but intentionally excludes remote
pump and brewing actions from API v1. This PRD introduces API v2 for physical
extraction control while leaving v1 temperature-control behavior available during
migration.

The owner reports that a second FOTEK SSR-40 DA is installed on the pump wiring
and working, with its active-high input connected to GPIO10. The original pump
switch remains a series hard cutoff. This is owner-provided hardware context, not
a substitute for the supervised SSR, wiring, enclosure, reset, and failure-mode
evidence required by `docs/SAFETY.md`.

## Requirements

### Profiles and mobile behavior

- Present five extraction selections: built-in Manual plus four stable custom
  profile slots whose editable names are not their identifiers.
- Give each custom slot an immutable, unique lowercase slug identifier. Profile
  names accept only ASCII letters and numbers and contain 1 through 12
  characters; this bound is intentionally small and negligible for ESP32 NVS.
- Seed editable `Classic30` (0 seconds pre-infusion, 0 seconds soak, 30 seconds
  main extraction) and `Pre5Soak5` (5 seconds pre-infusion, 5 seconds soak,
  25 seconds main extraction); leave two slots empty.
- Each profile contains a bounded name and whole-second pre-infusion pump-on,
  soak pump-off, and main extraction durations.
- Zero pre-infusion and zero soak represents a profile without pre-infusion.
- The total wall-clock profile duration must not exceed 60 seconds.
- Store the mobile profile set locally. Edits never synchronize automatically.
- Provide an explicit `Export to machine` action that atomically overwrites the
  complete ESP32 custom profile set.
- Reject export while extraction is active. Persistence failure must retain the
  previous complete machine set.
- Show when mobile and machine profiles differ and block starting the selected
  custom profile until the current mobile set is successfully exported.
- Keep distinct Start and Stop buttons visible on the primary extraction UI.
- Show acknowledged extraction status, phase, selected profile, elapsed and
  remaining time, and the ESP32 pump command as only `running` or `off`, without
  implying measured flow, switch position, current, or physical pump state.

### Extraction behavior

- A custom profile runs pre-infusion pump-on, optional soak pump-off, and main
  extraction pump-on, then returns to idle with the pump command off.
- Manual starts the pump until Stop, reset, failure, or a mandatory 60-second
  firmware cutoff.
- Firmware continues an acknowledged extraction when the phone disconnects or
  the app closes.
- Start uses a client-generated idempotency key. Retrying the same key returns
  the same extraction without resetting timing; a different Start while active
  is rejected as a conflict.
- Stop is idempotent and returns the acknowledged idle state even if extraction
  already ended.
- Extraction may start and continue regardless of brew/steam mode, readiness,
  or temperature/heater fault. Existing heater faults retain their heater
  fail-off behavior but do not stop the pump.
- Pump-control failure, Stop, Manual cutoff, profile completion, reset, or power
  loss stops extraction. Boot always initializes GPIO10 off and never resumes a
  prior extraction.

### Contract, persistence, and firmware

- Define API v2 first in `packages/protocol/openapi.yaml`, with strict request,
  response, profile, extraction-state, and conflict shapes.
- Retain API v1 as temperature-control-only during migration.
- Provide authenticated API v2 operations to read the machine profile set,
  atomically replace it while idle, start Manual or a persisted profile, and
  stop extraction.
- Mirror the contract in strict Zod schemas for mobile and simulator, and
  validate all inputs independently in firmware C++.
- Persist only the four custom profile slots atomically in ESP32 NVS. Active
  extraction state and idempotency state are volatile and reset to idle on boot.
- Implement extraction as a firmware-owned monotonic state machine whose pump-on
  deadlines cannot be extended by HTTP, display, profile persistence, mutex
  contention, or phone connectivity.
- Implement a dedicated fail-off pump output abstraction. GPIO configuration or
  write failure ends extraction, commands `off`, and returns the existing
  internal-error behavior without introducing pump-specific physical-status or
  fault codes. `running` and `off` describe firmware's GPIO command only and
  must not be presented as confirmed physical operation or de-energization.
- Keep pump and heater timing independent. Profile persistence must occur only
  while extraction is idle and outside real-time pump deadlines.

### Hardware and safety boundary

- Use the same active-high direct 3.3 V SSR-drive design as heater GPIO20 and
  document GPIO10 boot/reset behavior, the original series cutoff switch, and
  the absence of switch/current feedback.
- Preserve the existing rule that software commands and API responses do not
  prove physical output state; an SSR may fail short.
- Permit contract, simulator, mobile, host-test, target-build, and supervised
  low-voltage work before energized acceptance.
- Keep unresolved relevant security and mains-safety findings visible. Energized
  acceptance requires separately recorded supervised evidence under
  `docs/SAFETY.md`, including pump-load SSR suitability, wiring, enclosure,
  cutoff operation, reset behavior, and failure cases.

## Implementation Overview

1. Define the minimum API v2 profile and extraction contract needed to keep the
   mobile design aligned with the eventual wire behavior.
2. Build the mobile profile editor and extraction interface first, preserving
   the current app's visual language and using its debug mode with injected
   deterministic state for owner review and explicit design approval.
3. After design approval, add deterministic simulator profile persistence and
   extraction timing, then connect the approved mobile flow to API v2.
4. Add firmware profile storage, fail-off GPIO10 output, and extraction state
   machine independently of heater timing.
5. Align OLED and public documentation, then complete automated, low-voltage,
   and separately approved supervised hardware validation.

## Acceptance Criteria

- [ ] API v1 remains available and temperature-control-only while API v2 exposes
  strict authenticated pump and profile operations.
- [ ] Manual and exactly four stable custom slots are presented; the two defined
  examples are seeded and editable.
- [ ] Mobile, simulator, and firmware require unique immutable slug IDs, accept
  only 1–12 ASCII alphanumeric profile names, and reject invalid slot sets,
  non-integer durations, invalid phase combinations, and profiles longer than
  60 seconds.
- [ ] Export replaces all machine custom profiles atomically, is rejected while
  active, and preserves the prior set after persistence failure.
- [ ] An unsynchronized selected profile cannot start until export succeeds.
- [ ] No-pre-infusion and pre-infusion/soak/main profiles transition at their
  deterministic monotonic deadlines and finish with the pump command off.
- [ ] Manual Stop works and firmware commands the pump off at 60 seconds without
  requiring an app connection.
- [ ] Same-key Start retries do not restart timing; competing Start requests are
  rejected; Stop is idempotent.
- [ ] App disconnection does not interrupt an acknowledged extraction, while
  reset or power cycle boots idle, commands GPIO10 off, and never resumes it.
- [ ] Temperature/heater faults do not stop extraction, while pump-control
  failure ends it through existing internal-error handling; the UI exposes only
  the `running`/`off` GPIO command and never claims confirmed physical state.
- [ ] The mobile UI always provides profile selection and distinct Start/Stop
  controls, preserves the current app design language, supports owner review in
  debug mode before backend implementation, and displays only acknowledged live
  extraction state outside that explicit preview mode.
- [ ] Protocol/OpenAPI checks, affected Bun tests and typechecks, firmware host
  tests, and the pinned ESP-IDF target build pass.
- [ ] Supervised low-voltage tests verify GPIO10 startup-off, phase timing,
  failure handling, cutoff, reset, and timer-wrap behavior with the load
  disconnected.
- [x] Energized acceptance records separate owner-reported Human evidence for the exact
  pump, SSR, GPIO drive, series switch, wiring, enclosure, and failure cases and
  does not hide unresolved relevant review findings.
