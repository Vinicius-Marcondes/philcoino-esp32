# PRD-006 — Dashboard continuity and daily temperature history

Status: DRAFT — Implemented, Human acceptance pending

## Summary

Keep the last acknowledged Dashboard state visible when the mobile app is
temporarily backgrounded, refresh it immediately on return without presenting
it as live, and persist the current local day's foreground temperature samples
for graph restoration and CSV export.

## Goals

- Remove the blank reconnect experience after ordinary app minimization.
- Preserve acknowledged-state and mutation-cancellation safety boundaries.
- Restore today's graph across mobile sessions and make sampling gaps explicit.
- Export every stored current-day sample through the native share sheet.

## Non-Goals

- Continuous background polling, interpolation, notifications, or cloud sync.
- Device-side history, firmware/API changes, or history across prior days.
- Using recorded data to influence firmware control, targets, readiness, or
  faults.

## Context

The Dashboard polling session already supported non-clearing pause/resume, but
the AppState hook used stop/start, which cleared the snapshot and connection
presentation on every foreground transition. The temperature curve was bounded
component state and therefore disappeared on disconnect, remount, or process
restart.

Mobile operating systems do not guarantee one-second local HTTP work in the
background. This feature therefore records only validated foreground polls and
shows inactive or disconnected intervals as gaps.

## Requirements

- Pause and abort active Dashboard work when the app leaves the foreground;
  retain the last acknowledged snapshot and resume polling immediately.
- Mark retained data as refreshing and prevent all firmware mutations until a
  newly validated combined snapshot arrives.
- Treat a background-interrupted mutation as unacknowledged, ignore late
  responses, and retain existing idempotency-key recovery behavior.
- Store device ID, UTC capture time, firmware uptime, boiler temperature,
  targets, mode, active target, heater permission/command, status, and fault for
  each validated foreground poll.
- Keep only the current local calendar day, scope history to the selected
  device, clear its rows when forgotten, and preserve gaps and uptime resets.
- Provide detailed Live and downsampled Today graph views while exporting all
  raw stored rows in chronological RFC 4180 CSV.
- Keep storage/export failures non-blocking and exclude bearer tokens and
  network addresses from history and exports.

## Implementation Overview

1. Coordinate AppState through testable polling/mutation pause/resume lifecycle
   state and explicit Dashboard freshness.
2. Persist current-day rows through an injected SQLite repository with
   in-memory web/test adapters.
3. Rehydrate and render time-based Live/Today curves without connecting gaps.
4. Generate a temporary UTF-8 CSV, open the native share sheet, and delete the
   temporary file after completion.

## Acceptance Criteria

- [x] Backgrounding stops requests without clearing the acknowledged snapshot.
- [x] Foregrounding refreshes immediately and controls remain disabled until a
  valid response.
- [x] History survives app remount/restart during the same local day.
- [x] Offline/background intervals and ESP32 uptime resets are not interpolated.
- [x] Live and Today views remain available when current machine state is
  unavailable.
- [x] CSV contains every raw stored current-day row and no credentials/address.
- [x] Mobile tests, typecheck, lint, Expo SDK 54 config, and web export pass.
- [ ] Human confirms minimize/restore, force-close/reopen, midnight rollover,
  graph readability, and Files/AirDrop export on an iPhone.

## Open Questions

- Human visual and native share-sheet acceptance remains pending.
- Retention beyond the current local day remains intentionally deferred.
