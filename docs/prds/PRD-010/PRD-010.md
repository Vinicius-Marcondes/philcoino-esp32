# PRD-010 — Import extraction profiles from ESP32

Status: Complete — Human Accepted 2026-07-21

Approved: 2026-07-21

## Summary

Let the mobile app read the selected ESP32's complete four-slot extraction
profile set, review the differences, and explicitly replace the app-wide local
profile set. The workflow uses the existing authenticated API v2 profile read;
it does not change firmware state or the wire contract.

## Goals

- Add an explicit `Import from machine` action beside the current export flow.
- Fetch and validate the latest machine set before showing changes.
- Protect local profiles with a slot-by-slot review and confirmation.
- Make local profile persistence and remote profile loading recoverable and
  deterministic.

## Non-Goals

- Partial-slot merging, automatic synchronization, undo history, or per-machine
  mobile profile storage.
- Protocol, simulator model, firmware, extraction timing, or safety-loop changes.
- Resolving the separately tracked encrypted pairing/session migration.

## Context

API v2 already exposes authenticated `GET /api/v2/profiles`, and the app already
loads the acknowledged machine set for synchronization checks. The current UI
only copies profiles from mobile to ESP32. Review findings MOB-003 and MOB-004
also show that profile reads do not recover after an initial transient failure
and local saves can complete out of order; import depends on both paths.

## Requirements

- Import always reads a fresh, strictly validated complete machine profile set.
- When sets differ, show every changed immutable slot with local and machine
  values and require confirmation before replacing local storage.
- When sets match, report that no import is needed without writing storage.
- Publish imported profiles only after the complete local persistence write
  succeeds; failures preserve the previous local set.
- Keep all local profile writes serialized under one owner so stale completions
  cannot replace newer visible or persisted state.
- Load local and machine profiles independently, retain either successful value,
  retry machine reads on reconnect/focus and explicit request, deduplicate reads,
  and cancel stale publication on unmount.
- Disable import while extraction is active, connectivity is not live, or a
  conflicting profile operation is pending. Cooldown does not block this
  phone-only operation.
- Cancel review when its local source revision becomes stale, the app leaves the
  active foreground/focus lifecycle, or connectivity is lost.
- Preserve app-wide mobile profile storage, slot identifiers, strict profile
  validation, and English/Portuguese accessibility behavior.

## Implementation Overview

1. Centralize local profile writes and recoverable machine-profile reads.
2. Add a fresh-read, compare, review, cancel, and confirm import state flow.
3. Extend the Profiles sync presentation and debug behavior.
4. Add race/recovery/UI tests and align architecture and review documentation.

## Acceptance Criteria

- [x] Import fetches the latest complete machine set and reports matching sets
  without a local write.
- [x] Every changed slot is reviewed before the complete local set is replaced.
- [x] Fetch, validation, cancellation, or storage failure leaves local profiles
  unchanged and gives retryable feedback.
- [x] Concurrent editor/import writes cannot persist or publish stale state.
- [x] Profile controls recover after an initial remote failure without remounting.
- [x] Import cannot run during active extraction or offline/stale state and does
  not mutate ESP32 profiles.
- [x] Debug mode, portrait/landscape layouts, accessibility, and English and
  Portuguese copy support the complete workflow.
- [x] Affected mobile tests, typecheck, lint, and simulator regression checks
  pass.
