# PRD-005 — Firmware API codec decomposition and adversarial validation

Status: Complete — Human Accepted 2026-07-18

Approved: 2026-07-17

Accepted: 2026-07-18

## Summary

Refactor the firmware HTTP API implementation from the current monolithic
`components/networking/src/api.cpp` into small, host-testable owners for route
and authentication policy, JSON syntax handling, per-domain request parsing and
validation, response serialization, and controller/storage orchestration.

The first refactor must preserve the current API v1 and v2 wire behavior. Before
moving code, it must characterize accepted and rejected inputs, status codes,
error bodies, authentication challenges, and deterministic response payloads.
Each extraction must retain independent C++ contract validation and add
malformed-input property/fuzz coverage under host sanitizers.

The work is incremental architectural maintenance, not a protocol redesign. It
must record ESP32-C3 firmware size, static RAM, request-path stack use, compile
warnings, and pinned ESP-IDF 6.0.2 compatibility at every extraction stage.

## Goals

- Remove `api.cpp` as a single maintenance hotspot without changing current
  public or authenticated API behavior during the initial refactor.
- Give JSON syntax, domain codecs, routing/authentication, and orchestration
  separate single-purpose, host-testable ownership.
- Make malformed and adversarial request behavior explicit, deterministic, and
  sanitizer-backed rather than relying only on route-level examples.
- Preserve firmware's independent implementation of the language-neutral
  OpenAPI contract and strict rejection of untrusted input.
- Keep future API versions and the separate FW-005 authenticated-session work
  from depending on one bearer-specific, version-entangled codec unit.
- Detect target resource regressions at the stage that introduces them rather
  than only after the complete refactor.

## Non-Goals

- Adding, removing, renaming, or versioning endpoints, fields, error codes, or
  authentication schemes under the behavior-preserving refactor.
- Implementing FW-005 encrypted sessions, device identity, credential rotation,
  revocation, certificate pinning, or downgrade protection.
- Changing temperature, extraction, compensation, cooldown, persistence,
  idempotency, synchronization, output-command, or fault semantics.
- Replacing the current codec with a third-party JSON, HTTP, schema, fuzzing, or
  testing dependency without separate explicit approval.
- Sharing TypeScript schema implementation with firmware or generating C++
  runtime code from Zod. Firmware must continue validating independently.
- Treating host sanitizers, fuzzing, contract captures, or a successful target
  build as physical heater/pump, timing, wiring, or mains-safety evidence.
- Opportunistically refactoring unrelated firmware findings in the same change.

## Context

FW-013 in `docs/reviews/FIRMWARE_CODE_REVIEW.md` identified the then 1,281-line
hand-written API codec as a maintainability hotspot and required a dedicated PRD
outside the FW-001–FW-012 remediation batch. The file is now approximately
1,376 lines and combines a custom JSON object parser, nested-array handling,
domain validation, serializers, bearer comparison, route dispatch, workflow
locking, persistence transactions, and controller coordination.

The firmware currently serves two public routes and twelve protected API v1/v2
routes. `EspNetworkServer` owns ESP-IDF HTTP adaptation, the 512-byte
authorization-header limit, the 1,024-byte request-body limit, socket timeouts,
and the absolute body deadline. FW-012 additionally requires protected routes
to reject missing or malformed authorization before reading their bodies.
`FirmwareApi` then independently checks access, parses requests, coordinates the
single workflow boundary, calls storage/controllers, and serializes responses.

The existing host suite covers representative success, validation, failure,
workflow, and response-capture paths, and TypeScript validates firmware captures
against shared protocol schemas. It does not yet provide a complete route/input
characterization, isolated codec tests, property tests, sanitizer-backed fuzz
targets, or per-stage target resource evidence.

FW-005 remains a release-blocking cross-layer security change. The approved
sequence is to complete PRD-005 before starting FW-005: first extract JSON
syntax and serialization, then the per-domain codecs, and consolidate the
current route/access boundary last. After PRD-005 is accepted, FW-005 may replace
that cleaner boundary through its own approved cross-layer work. PRD-005 must
encapsulate today's bearer behavior without inventing speculative future session
interfaces, and the two initiatives must not modify routing/authentication
concurrently.

## Requirements

### Behavior baseline and compatibility freeze

- Before moving production code, create a machine-readable characterization
  matrix for every supported method/path pair, both public routes, all protected
  API v1/v2 routes, unsupported methods, and unknown paths.
- Characterization must cover missing, malformed, wrong, and valid
  authorization; `WWW-Authenticate` behavior; status codes; stable error codes
  and messages; and deterministic response bodies for representative state.
- For each body-bearing route, characterize empty input, valid field-order and
  whitespace variations, duplicate and unknown fields, wrong JSON types,
  malformed strings/escapes/numbers, nested objects/arrays, boundary values,
  constraint violations, trailing data, and route-specific conflicts/failures.
- Preserve byte-for-byte deterministic bodies for existing golden states,
  including field order and numeric formatting, unless the authoritative
  OpenAPI contract already proves that an existing capture is invalid.
- Run the existing firmware capture validator against the authoritative strict
  schemas before the first extraction and after every extraction.
- If characterization finds current firmware/OpenAPI drift or unsafe behavior,
  record it as a separate decision. Do not silently change the wire behavior or
  weaken the contract as part of this refactor.
- API v1 must remain temperature-control-only and unchanged. API v2 must retain
  its current strict workflow, replay, conflict, and combined-state shapes.

### Single-purpose firmware boundaries

- Separate low-level JSON syntax handling from all machine-domain knowledge.
  Its pure interface must accept bounded input and return explicit success or
  failure without controller, storage, output, ESP-IDF, or network access.
- Separate request parsing and validation by domain: temperature/mode/heater,
  profiles/extraction, cooldown, and shared identifiers/scalars. Domain parsers
  must return typed values and retain duplicate/unknown-field rejection.
- Separate response and error serialization from request parsing. Serializers
  must consume immutable snapshots/results and must not acquire locks, mutate
  controllers, perform persistence, or access the network.
- Give route metadata and access policy one authoritative owner that both HTTP
  registration/pre-body authentication and firmware dispatch can consume
  without duplicating method/path protection rules.
- Keep transport concerns in the ESP-IDF adapter: header/body bounds, absolute
  receive deadline, timeout count, request-body acquisition, response
  transmission, and bearer-challenge header emission.
- Keep workflow orchestration in explicit handlers that own lock acquisition,
  snapshot timing, controller calls, persistence prepare/save/adopt/rollback,
  idempotency decisions, and mapping domain outcomes to HTTP responses.
- Keep pure codec and policy modules buildable in the native C++ host suite with
  no ESP-IDF dependency. The public networking header must expose only the
  types/interfaces needed by callers.
- Perform the decomposition in small stages. Each stage must compile, pass all
  affected tests, preserve the behavior baseline, and receive review before the
  next ownership boundary is moved.

### Security, timing, and safety invariants

- Preserve strict duplicate- and unknown-field rejection, finite-number rules,
  integer/range constraints, profile slot/order rules, idempotency-key rules,
  error taxonomy, and version-specific response shapes.
- Preserve public/protected route boundaries and FW-012's rule that an
  unauthorized protected request is rejected before its body is read.
- Preserve the 512-byte authorization-header bound, 1,024-byte body bound,
  two-second absolute body deadline, maximum timeout count, and bounded server
  socket timeouts. Pure codec entry points must never assume a larger input.
- Preserve constant-time length-aware bearer comparison until FW-005 replaces
  the authentication design through its own approved cross-layer work.
- No codec function may directly command the heater or pump, persist NVS data,
  change a controller, or own a workflow lock.
- Preserve the single bounded workflow synchronization boundary, persistence
  outside real-time locks, transactional target/profile acknowledgement, and
  all existing fail-off, inhibit, deadline, fault, replay, and command-state
  semantics.
- Parsing, serialization, logging, or error construction must not expose bearer
  tokens, Wi-Fi credentials, device private material, or future session secrets.
- The refactor must not make Stop/state traffic wait on new unbounded work or
  introduce additional work inside real-time workflow critical sections.

### Property, fuzz, and sanitizer coverage

- Add direct unit tests for the generic JSON syntax boundary and every
  per-domain request/response codec, independent from controller orchestration.
- Add deterministic property tests covering valid-object field permutations,
  allowed whitespace, boundary values, duplicate/unknown field injection,
  scalar type substitution, truncation at every byte position, and malformed
  string/escape/number/composite forms.
- For every body-bearing route, assert that any rejected payload leaves
  controller, storage, idempotency, fault, heater-command, and pump-command state
  unchanged.
- Provide host fuzz entry points for the generic JSON boundary and each distinct
  domain parser. Seed them from protocol fixtures, firmware captures, existing
  regression cases, and minimal valid/invalid payloads.
- Keep the entry points engine-neutral and compatible with
  `LLVMFuzzerTestOneInput` so an already-available coverage-guided engine can
  consume them later without changing production codec code.
- Fuzz properties must include: no crash, undefined behavior, leak, hang, or
  out-of-bounds access; deterministic classification for the same input; no
  acceptance of unknown/duplicate fields; and no mutation from rejected input.
- Provide a repository-owned deterministic corpus/mutation runner as the
  mandatory local fuzz/property execution path. Compile and run it together
  with the codec/property suites under AddressSanitizer and
  UndefinedBehaviorSanitizer.
- Coverage-guided libFuzzer execution is optional when an already-approved
  environment provides its runtime. Its absence does not block PRD-005, and this
  PRD does not authorize installing a compiler, fuzzing runtime, or dependency.
- Keep a short reproducible sanitizer-backed mutation command suitable for
  routine verification and document the optional coverage-guided command.
  Every discovered crash or semantic mismatch from either path must become a
  permanent minimized regression seed.

### Target compatibility and resource evidence

- Establish a release-configuration baseline on ESP32-C3 with pinned ESP-IDF
  6.0.2 before the first production extraction.
- At baseline and after every extraction stage, record the application image and
  partition fit, `.text`, `.rodata`, `.data`, and `.bss` sizes, relevant heap
  observations, and HTTP/request-path stack high-water evidence under the same
  representative maximum-size requests.
- Apply these cumulative limits against the pre-refactor release baseline at
  every stage and at final integration:

  - application image: no more than `2%` larger;
  - combined `.data + .bss`: no more than `1 KiB` larger;
  - peak request heap consumption: no more than `1 KiB` larger under the same
    characterized request scenarios;
  - HTTP task unused stack: at least `1,536 bytes` and no more than `512 bytes`
    below the baseline high-water margin.
- Retain the current `6,144-byte` HTTP task stack configuration. Do not increase
  it merely to satisfy the margin; a measured need requires a separate reviewed
  exception.
- Treat an unexplained resource regression, reduced stack margin, new compiler
  warning, partition overflow, or target incompatibility as a stage blocker.
  Do not hide it by weakening warnings, removing tests, or increasing target
  resources without review.
- Native host targets must retain strict C++17 warning checks. The complete
  firmware must build for ESP32-C3 with the repository-pinned configuration and
  no new warnings attributable to the refactor.
- Resource and target-build results are software/integration evidence only and
  must not be described as energized hardware or safety validation.

### Documentation and supervised execution

- Split this PRD into supervised, reviewable tasks only after the PRD is approved.
  The task sequence must keep characterization, each extraction boundary,
  adversarial testing, target measurements, and final integration evidence
  independently reviewable.
- Complete PRD-005 before starting FW-005 changes to routing/authentication.
  Within PRD-005, extract JSON/serialization first, per-domain codecs second,
  and current route/access policy last; do not introduce a speculative session
  abstraction for FW-005.
- Keep `docs/ARCHITECTURE.md`, firmware development instructions, protocol
  outlines, review status, and verification boundaries aligned with the final
  owners and commands.
- Do not mark FW-013 complete from smaller files alone. Closure requires the
  behavior matrix, adversarial coverage, sanitizer evidence, per-stage resource
  records, pinned target build, and final independent contract validation.
- Preserve FW-005 and all hardware/safety blockers as separate visible work.

## Implementation Overview

1. Freeze current behavior with route/input characterization, golden responses,
   capture validation, and an ESP32-C3 resource baseline.
2. Extract and directly test the bounded generic JSON syntax/encoding boundary,
   then add deterministic malformed-input properties and sanitizer harnesses.
3. Extract typed per-domain request parsers and immutable response serializers
   one domain at a time, re-running the complete compatibility and resource
   gates after each stage.
4. Consolidate route metadata/access policy while preserving pre-body
   authentication, then leave controller/storage coordination in explicit thin
   orchestration handlers.
5. Run the full seeded fuzz/property campaign, minimize and preserve regressions,
   validate all firmware captures independently against the shared contract,
   and complete the pinned ESP32-C3 build/resource comparison.
6. Update architecture, development, protocol, safety-boundary, and review
   documentation; retain FW-005 and physical validation as separate work.
